CodeFixEnv — concept walkthrough

concept 1 of 6

What even is an "environment"?

Forget code for a sec. Imagine a driving test examiner. They give you a task ("park the car"), you attempt it, and they give you a score. That's it. That's an environment.

In CodeFixEnv — you're building that examiner. The AI agent is the student. The task is "fix this broken code."

Examiner (your env) → gives agent: buggy code + failing test Agent → submits: fixed code Examiner → grades it: reward 0.0 → 1.0 repeat until solved or max attempts hit

Your job is to build the examiner, not the student.

concept 2 of 6

The 3 API methods — reset, step, state

Every OpenEnv environment speaks 3 words. That's the whole spec.

reset() → "New exam. Here's your buggy code." Returns the starting observation (what agent sees) step(action) → "Agent submitted a fix. Let me grade it." Returns new observation + reward score state() → "What's happening right now in the exam room?" Returns snapshot (for graders + debugging)

Judges run these 3 methods automatically. If any of them break — you're disqualified. That's why getting them right is everything.

concept 3 of 6

Why "code fixing" is a perfect choice

The rules say: must be a real-world task, not a game or toy. Code debugging is about as real-world as it gets — it's literally what Copilot, Cursor, and Devin do.

And it's objectively gradeable — either the test passes or it doesn't. No human judgment needed. The grader just runs the code.

Task 1 (easy) → syntax error / off-by-one bug Task 2 (medium) → logic bug, wrong output Task 3 (hard) → multiple bugs + edge cases

Difficulty progression = judges love this. It shows you thought about the design, not just threw something together.

concept 4 of 6

The reward signal — this is the soul of RL

Reward is how the agent learns. If you just give 0 or 1 (fail/pass), the agent gets no useful signal — it's flying blind. Partial rewards let it know "I'm getting warmer."

0.0Code won't even parse (syntax error)
0.25Runs but crashes at runtime
0.5Runs, main test fails (AssertionError)
0.75Main test passes, edge cases fail
1.0All tests pass — perfect fix

This partial reward design is what separates a good submission from an average one.

concept 5 of 6

What you're actually building — 7 files

The whole project is just 7 files. Each has one job.

models.pydefines what an "action" and "observation" look like
environment.pythe actual logic — tasks, graders, reward math
server.pywraps it in FastAPI so judges can hit HTTP endpoints
openenv.yamlconfig manifest — tells OpenEnv what your env is
inference.pythe LLM agent script that plays through your environment
Dockerfilepackages everything so HF Spaces can run it
README.mdpublic description — judges read this too
concept 6 of 6

What "passing Round 1" actually means

Judges run an automated checker. It does 5 things in order:

1. ping your HF Space URL → must return 200 2. validate openenv.yaml → must match spec 3. docker build your repo → must succeed 4. run inference.py → must finish, print [START][STEP][END] 5. call each task grader → must return score 0.0–1.0

If all 5 pass — you're through. The quality score (what gets you ranked higher) comes from how clean your reward logic is and how well-designed your tasks are.

You already have a strong concept. Now it's just execution.
